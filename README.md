# 企业服务云平台建设 ZhengQi
企业服务云平台建设，将围绕金融办、工商联、发改委、经信委、人社局、科技局、政务办等各部门需求，科学规划涉企服务和服务管理功能，搭建区企业综合服务平台门户包括区级各部门涉企的政务服务、企业生产经营的各类主题服务事项和涉企综合服务专栏，并支持服务智能搜索、服务推荐及服务数据统计能力。通过智能客服，给企业提供服务，AI助手具备链接企业服务入口功能，构建金融、政策匹配底层模型能力，能够嫁接到公众号与PC门户平台，提供根据标签匹配政策库、金融库能力。运营管理子系统作为企业码区级运营后台，融合区企服后台功能，需实现用户账号管理、诉求管理、服务统计、考核管理、企业码管理、运营指标管理、应用服务管理、政策兑现项目管理、服务管理等功能，以及各部门提出的其他运营管理需求，以满足区级部门的整体后台管理需要。通过汇集、融合各类相关数据，实现对企业的全量数据分析，画像，根据大数据发展局要求开发数据统计、分析、可视化等功能。
#1.2设计思路
建设企业服务云平台，将按照“ 1个统一集成平台 + N 个专业应用系统”、“全面成套、层次恰当、划分明确”的原则，以区企业综合服务平台门户、企业AI助手、企业码门户、运营管理子系统、数据底层服务等涉企服务为业务主线，以平台具备大容量、高处理能力、高可靠性、高稳定性、高扩展性、高效性、高开放性、高安全性、易维护性、易用性、先进性、可持续性为硬性指标，切实考虑企业服务云平台的实现技术，采用当前成熟、先进、适用的信息化技术架构。搭建一套功能完善、针对性强，满足各角色用户需求的业务应用功能体系架构。企业服务云平台采用模块化、构件化设计思想，采用“松耦合、易迭代”的开发思路，将系统划分为各个相对独立又相互配合的业务子系统、功能模块和功能构件，通过统一的接口和参数进行功能结构组合与信息传递，满足动态功能扩展和调整。
#1.3架构设计
本次企业服务云平台建设，其系统总体架构设计，由“6 层架构+4大体系”组成，六层架构包括基础设施层、数据资源层、应用支撑层、应用层、展现层、用户层，4大体系包括标准规范体系、运行管理体系、安全保障体系、政策法规制度体系组成。企业服务云平台，系统总体架构设计如下图所示：
1、基础设施层：主要包含各类网络环境、硬件基础设施和安全环境三大部分。主要包括存储、服务器、交换机、防火墙等硬件基础设施。以上相关基础设施资源由政务云提供服务。
2、数据资源层：主要包括基础数据库、业务数据库、统计分析库。基础数据库主要包括企业信息库、机构信息库、用户信息库等。业务数据库主要包括金融服务库、政策服务库、人才服务库等。统计分析库主要包括金融统计库、需求统计库、人才统计库、用户统计库等。通过数据库软件进行结构化与非结构化数据的存储，同时实现读写分离的主从配置，保障数据库的读写效率及稳定性，为应用支撑层和业务应用层提供数据支撑。
3、应用支撑层：应用支撑层包含基础性、底层性的应用系统支撑功能主要包括基础应用服务、支撑软件和业务模型服务。业务模型服务包括金融产品匹配模型、政策推荐模型、人才岗位匹配模型。
4、业务应用层：搭建集金融、政策、诉求直达、科创、人才、商会、助企服务等余一体的业务管理功能，实现金融服务领域、政策服务领域、科创服务等领域业务的数字化在线管理与数字化服务。
5、展现层：包含为政府工作人员以及辖区金融机构、企业、商会等，以及运营管理人员提供PC端服务、移动端服务。满足各用户角色在不同场景下使用系统，主要为政府工作人员、金融机构等提供PC端桌面服务，为企业提供移动端服务。
6、用户层：主要包括政府工作人员以及辖区金融机构、企业、商会、运营管理人员等。为政府端提供金融管理、政策管理等功能。为金融机构提供金融产品申请受理、融资需求受理等功能。为企业提供金融产品申请、政策申请等功能。为运营管理人员提供消息通知、运营统计、用户账号管理等功能。
7、四大体系：将严格遵循国家层面、行业层面地方层面相关信息化标准规范、运行管理体系、安全保障体系、政策法规制度体系与要求进行建设，保障智慧园区综合管理与服务平台安全、高效运行。

1.4技术路线
1.4.1J2EE技术
系统采用J2EE为核心的三层架构，支持jdk1.6以上版本。J2EE是一种利用Java 2平台来简化应用软件系统开发、部署和管理相关的复杂问题的体系结构。J2EE技术的基础就是核心Java平台或Java 2平台的标准版，具有“编写一次、随处运行”的特性，方便存取数据库的JDBC API、CORBA技术以及能够在Internet应用中保护数据的安全模式等，同时还提供了对 EJB（Enterprise JavaBeans）、Java Servlets API、JSP（Java Server Pages）以及XML技术的全面支持。
J2EE体系结构提供中间层集成框架用来满足无需太多费用而又需要高可用性、高可靠性以及可扩展性的应用的需求。通过提供统一的开发平台，J2EE降低了开发多层应用的费用和复杂性，同时提供对现有应用程序集成强有力支持，完全支持Enterprise JavaBeans，有良好的向导支持打包和部署应用，添加目录支持，增强了安全机制，提高了性能。
1.4.2React前端技术
前端框架采用主流的React。React是一个用于构建用户界面的 JavaScript 库，React主要用于构建UI。你可以在React里传递多种类型的参数，如声明代码，帮助你渲染出UI，也可以是静态的HTML DOM元素，也可以传递动态变量，甚至是可交互的应用组件。具有如下特点：
1.React采用声明式设计，可以轻松描述应用；
2.React通过对DOM的模拟，最大限度地减少与DOM的交互，高效开发；
3.React可以与已知的库或框架很好地配合，灵活性高；
4.使用组件化开发方式，符合现代Web开发的趋势；
5.技术成熟，社区完善，配件齐全，适用于大型Web项目（生态系统健全）；
6.使用方式简单，性能非常高，支持服务端渲染；
1.4.3Redis缓存技术
本方案采用Redis作为数据缓存系统。Redis是Nosql数据库中使用较为广泛的非关系型内存数据库，Redis内部是一个key-value存储系统。它支持存储的value类型相对更多，包括string(字符串)、list(链表)、set(集合)、zset(sorted set –有序集合)和hash（哈希类型，类似于Java中的map）。
Redis有三个主要特点，使它优越于其它键值数据存储系统：
1.Redis将其数据完全保存在内存中，仅使用磁盘进行持久化。
2.与其它键值数据存储相比，Redis有一组相对丰富的数据类型。
3.Redis可以将数据复制到任意数量的从机中。
4.Redis还具有以下优点：
1)非常快，Redis每秒可执行大约110000次的设置(SET)操作，每秒大约可执行81000次的读取/获取(GET)操作。
2)支持丰富的数据类型，Redis支持开发人员常用的大多数数据类型，例如列表，集合，排序集和散列等等。这使得Redis很容易被用来解决各种问题，因为，我们知道哪些问题可以更好使用地哪些数据类型来处理解决。
3)操作具有原子性，所有Redis操作都是原子操作，这确保如果两个客户端并发访问，Redis服务器能接收更新的值。
Redis在互联网大数据平台有着广泛的应用，主要被用来缓存热点数据，避免海量请求压垮数据库，同时，可以提升服务节点的响应速度和并发量。随着数据量的增多，由于Redis是占用单台物理机或虚机的内存，内存资源是有限的，要动态地扩容缩容，可以用到Redis集群。

1.4.4基于构建技术的系统搭建模式
面向构件技术整合了传统的构件技术、模型驱动技术和面向服务技术。构件的意义超越了技术层面的代码集合，而是真正做到与业务层面的内容相映射。构件将成为表达业务需求的最小单元，在特定行业中，通过积累可以形成相对完备的客户业务需求构件库，以构件组装的形式快速搭建客户的应用；以构件修改和构件增减的方式快速满足客户业务需求的变化。其次，从构件的技术实现角度来说，可以用四个要素来陈述软件构件模型：构件本身、该构件的插口、构件与其他构件协作的能力、以及构件的使用者。
面向构件的开发是一种软件开发手段，在开发周期的不同阶段和不同方面，包括需求分析、结构、设计、建立、测试、上线、支撑性技术架构、项目管理等，都以构件为基础，同时为了使构件可随时用于组装，这些构件必须作为项目的零件来建造。
本项目将采用面向构件的系统分析与设计方法，以减少代码量和复杂性，增加构件的可重用性，从而增加系统的灵活性和扩展性，使得系统能够稳定运行。
1.4.5云计算
项目建设借助云计算（cloudcomputing）来实现基于互联网的相关管理与服务的增加、使用和交付，通过互联网来提供动态易扩展且经常是虚拟化的资源。本项目建设是一个复杂的系统工程，涉及海量信息的处理，包含多对象、多层次的综合应用和服务，多个应用系统之间存在信息共享、交互的需求。云计算技术可以使得这些资源和服务通过网络随时获取，按需使用，随时拓展，具有超大规模、虚拟化、快速弹性、可靠安全等特点。在本项目建设中应用云计算技术，采取云计算平台构架来搭建平台的技术设施、信息资源、各种应用和服务，将在很大程度上降低平台的建设成本。
第三章 详细设计方案
1.1信息资源规划设计
1.1.1规划目的
信息资源规划是指对企业服务云平台建设项目所需要的资源信息，从采集、处理、传输到使用的全面规划。旨在根据企业服务与管理需求，规划一套全面、科学、合理的信息资源数据库，满足业务功能应用需求与统计分析要求。本次项目的信息资源规划，将以金融服务管理、政策服务管理、诉求直达服务管理、科创服务管理、人才服务管理等涉企服务的应用为导向，对各版块业务资源进行梳理，合理规划数据资源，并可灵活适配、按需扩容，为涉企服务相关应用提供有力支撑。
1.1.2业务职能分析
职能域是功能划分的抽象，不能照搬现有业务处室名称，因为不同处室之间存在业务交叉。此过程需建立三个模型，包括职能域模型、业务过程模型、业务活动模型。职能域模型是对职能划分的抽象，不是组织机构的划分；业务过程模型，包括对业务过程的识别、命名和定义；业务活动模型是对业务过程进行分解，分解至不可再分解的最小功能单元。
根据对企业服务工作的分析，涉企业务可划分为：金融服务管理、政策服务管理、诉求直达服务管理、科创服务管理、人才服务管理、商会服务管理、助企服务管理及企业码应用。
1.1.3资源分类体系
数据资源建设主要是对数据从采集、传输、存储、更新、服务全过程的工作范围、工作内容和标准规范进行限定。通过对企业服务工作的分析，数据资源主要包括各职能部门内部产生的业务数据和企业产生的业务数据，形成全区统一的全要素涉企服务资源库，满足各职能部门内部调用和使用，满足企业、金融机构等主体使用。
根据企业服务现状和要求，对涉企金融服务、政策服务、诉求直达、科创在线、人才在线、商会服务、助企服务等主要业务管理内容进行归类整理，形成全要素涉企服务资源库，包括基础数据库、业务数据库（金融数据库、政策服务数据库、诉求服务库、科创服务库、人才服务库等）、统计分析库。
1.1.4数据库分类设计
通过对本次企业服务云平台建设项目所需信息资源的深入分析，将规划并建成基础数据库、业务数据库、统计分析库。
1.基础数据库
建设满足企业服务应用的基础数据库，主要包括企业信息库、金融机构库、市场服务主体库、组织架构信息库、组织用户信息库、角色权限库、字典库等，为金融服务、政策服务、诉求直达、科创在线、人才在线、商会服务、助企服务等应用服务提供基础数据支撑。
2.业务数据库
业务数据库是在业务办理过程中产生，并直接支持各项业务办理和对外服务的信息资源库，将围绕涉企服务业务，建立金融服务库、政策服务库、诉求直达服务库、科创服务库、人才服务库、商会服务库、助企服务库等，满足业务运行与应用需要。
3.统计分析库
根据平台各业务版块产生的业务数据，按照业务统计口径及规则，建立金融统计分析库、政策统计分析库、诉求直达统计分析库、科创服务统计分析库、人才服务统计分析库、助企服务统计分析库等，满足工作人员的日常工作需要，全面提升工作人员涉企服务工作效率。同时为满足运营管理人员关于平台的运营分析需要，构建用户统计分析库、进件数据分析库、订单分析库等。
1.2业务应用系统设计
1.2.1企业码管理子系统
企业码服务子系统，主要依托公众号为企业等用户提供企业码中心、扫码服务功能，详细功能设计如下：
1.2.1.1企业码中心
1.2.1.1.1快捷入口
集成扫码、示码、领码、企业名片、企业信息功能，企业、机构、政府等用户可快速访问并进入对应的应用功能。
1.2.1.1.2扫一扫
实现通过扫企业出示的企业码，可查看企业名片集成的相关信息，主要包括企业基本信息、投资信息、资产信息、法务信息。
1.2.1.1.3企业码
展示企业的企业码，通过扫码，扫码对象可查看企业的相关信息。同时提供企业名片、企业领码记录以及金融/政策智能匹配、智能建议与信贷建议、诉求直达、领取企业码、功能入口。
1.2.1.1.4在线领码
实现企业查询、选择需要领码的企业，填报并提交企业授权认证相关信息，审核通过，系统自动为企业生成并发放企业码。
1.2.1.1.5领码记录
企业统一维护管理本企业历次领取企业码的领码记录，可查看历次领码记录的企业名称、企业标识、领取时间信息。并可查看历次领码记录的当前状态。
1.2.1.1.6企业信息
企业统一维护管理本企业历次提交的企业认证信息记录，提供重新认证、完善资料、解榜、添加企业认证功能。
1.2.1.2扫码服务
服务一码通，企业、政府扫码畅享平台各项服务，实现掌上订阅，掌上申请，掌上办理等，体验涉企服务近在指间。
1.2.1.3企业信息管理
1.2.1.3.1领码审核
工作人员统一维护管理各企业提交的领码申请记录，提供多条件查询、查看详情、审核等通能。
1.2.1.3.2企业码管理

1.2.6.3用户账号管理
1.2.6.3.1企业用户管理
运营管理人员统一维护管理企业注册账号，提供多条件查询、查看详情、启用账号、停用账号功能。
1.2.6.3.2机构账号管理
运营管理人员统一维护管理发文机关、金融机构等的账号，提供多条件查询、查看详情、重置密码、账号信息修改、启用账号、停用账号、账号赋权功能。
1.2.6.3.3金融机构管理
运营管理人员统一维护管理金融机构信息，提供多条件查询、查看详情、编辑功能。
1.2.6.3.4企业信息管理
运营管理人员统一维护管理企业信息，提供多条件查询、查看详情、导出查询结果、下载企业分析数据。下载企业分析数据，实现按日期统计各园区的新增企业注册数，以Excel方式导出分析结果。
1.2.6.4角色权限管理
运营管理人员统一维护管理各机构的角色权限，提供多条件查询、查看详情、编辑、复制、删除功能。编辑，支持按功能权限、数据权限、字段权限配置角色权限。
1.2.6.5操作日志管理
运营管理人员统一维护管理各用户的操作日志、登录日志以及信息推送日志。其中操作日志提供多条件查询、查看详情功能，登录日志提供多条件查询功能，信息推送日志提供多条件查询功能。
1.3业务支撑系统设计
1.3.1操作系统
操作系统是所有应用的基础，本次项目操作系统将采用政务云平台提供的CentOS 7。
1.3.2数据库
本次数据库软件将采用PostgreSQL，PostgreSQL面向企业复杂SQL处理的OLTP在线事务处理场景，支持NoSQL数据类型（JSON/XML/hstore）。
1.3.3用户管理服务
提供统一用户注册、管理服务引擎，支撑多机构用户统一维护管理、统一授权、统一日志审计。
1.3.4消息服务
消息服务是一种高效、可靠、安全、便捷、可弹性扩展的分布式消息机制，能够帮助应用系统在组件上自由的传递数据、通知消息，构建松耦合系统。通常的消息服务支持队列模型和主题模型。队列模型支持一对一发送和接收消息；主题模型支持一对多发布和订阅消息，并且支持多种消息推送方式。
1.3.5接口服务
提供短信接口服务，支撑根据系统业务场景产生的消息，通过短信接口调用三方短信服务平台，向用户推送短信消息。
1.3.6数据服务
汇集、融合本平台各业务产生的各类相关数据，实现对企业的全量数据分析，画像，根据大数据发展局要求开发数据统计、分析、可视化等功能。
1.3.7模型服务
构建金融、政策匹配底层模型能力，通过嫁接到公众号与PC门户平台，提供根据标签匹配政策库、金融库能力，其企业推荐政策政策、推荐金融产品。
